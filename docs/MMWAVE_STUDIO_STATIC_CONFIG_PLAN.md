# mmWave Studio Static Config Plan (Dry-Run)

## Overview

The static config plan shows what ar1 calls would be needed to configure
the ADC, LVDS lanes, and data path on the AWR2944EVM — but does **not**
generate an executable Lua script.

This is a planning/preview tool for understanding how `capture.yaml`
fields map to mmWave Studio ar1 API calls.

## Usage

```bash
awr mmws static-plan capture.yaml
```

## What It Shows

| ar1 Call | capture.yaml Source | Purpose |
|----------|-------------------|---------|
| `ChanNAdcConfig` | `hardware.rx_enabled`, `hardware.tx_enabled`, `adc.bits`, `adc.is_complex` | Channel and ADC configuration |
| `LPModConfig` | `adc.is_complex` | Low-power mode selection |
| `DataPathConfig` | `adc.is_complex`, `adc.channel_interleave` | Data path interface |
| `LvdsClkConfig` | `adc.num_lvds_lanes` | LVDS clock configuration |
| `LVDSLaneConfig` | `adc.num_lvds_lanes` | LVDS lane mapping |

## Safety Checks

The plan applies the following safety checks:
- **RF LDO bypass**: Refused for now
- **Multiple TX at 0 dB backoff**: Warning for first bring-up safety
- **DCA Ethernet speed**: Informational warning if ≠ 1 Gbps
- **Python serial open**: Refused if mmWave Studio is expected to own the port

## Execution Status

All static config calls are currently marked `allowed_yet = False`.
No executable Lua script is generated by this command.

When static config execution is eventually enabled, it will follow
the same bridge pattern: Python generates Lua → user runs in Studio →
Lua writes result JSON → Python checks status.
